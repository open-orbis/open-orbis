# CV Extraction Quality Tests

Automated integration tests that evaluate how well the LLM extracts structured knowledge graph nodes from CVs. The tests run on every PR that touches the CV parsing pipeline and block merge if quality regresses.

## How it works

### Test flow

1. **Load CV fixture** — plain text extracted from a real CV (`tests/fixtures/*_cv.txt`)
2. **Run extraction** — call Claude CLI (`claude -p`) with the production system prompt to classify the CV into structured nodes (education, work_experience, skill, etc.)
3. **Compare against baseline** — fuzzy-match the predicted nodes against a baseline using the graph comparator
4. **Assert quality thresholds** — fail if metrics drop below minimums

### Baseline strategy

- **In CI**: the baseline is generated by running the extraction pipeline on the **main** branch. The PR's output is compared against it, making this a **regression test**.
- **Locally**: falls back to static golden reference files (`*_golden.json`) so you can run tests without extra setup.
- **Bootstrapping**: on the very first run (when `generate_baseline.py` doesn't exist on main yet), the golden files are used automatically.

### Metrics

The **graph comparator** (`tests/lib/graph_comparator.py`) performs greedy node matching per type using token-level Jaccard similarity with SequenceMatcher fallback. It computes:

| Metric | Description | Threshold |
|--------|-------------|-----------|
| **F1** | Harmonic mean of precision and recall | >= 0.70 |
| **Recall** | Matched nodes / baseline nodes | >= 0.60 |
| **Composite** | 0.5 × F1 + 0.5 × property similarity | >= 0.65 |

Per-type recall minimums:

| Node type | Min recall |
|-----------|-----------|
| education | 0.66 |
| work_experience | 0.55 |
| language | 1.00 |
| skill | 0.40 |

### Adding a new CV

Drop two files into `backend/tests/fixtures/`:

- `<name>_cv.txt` — plain text of the CV
- `<name>_golden.json` — expected nodes (used as local fallback)

The test auto-discovers all `*_cv.txt` files and parametrizes over them. No code changes needed.

## Running locally

```bash
cd backend
source .venv/bin/activate
pip install -e ".[dev]"

# Run the integration tests (requires claude CLI authenticated)
python -m pytest tests/integration/test_kg_quality.py -v -s -m integration
```

To generate baselines locally (optional):

```bash
python -m tests.integration.generate_baseline ./baselines
```

## CI workflow

The GitHub Actions workflow (`.github/workflows/cv-extraction-quality.yml`) triggers on PRs that modify:

- `backend/app/cv/**`
- `backend/app/graph/queries.py`
- `backend/tests/**`

### Two-phase execution

1. **Checkout main** — install dependencies, run `generate_baseline.py` to produce baseline JSON files in `/tmp/kg_baselines/`
2. **Checkout PR branch** — install dependencies, run pytest against the baselines

If the test fails, the workflow fails and the PR cannot be merged (when branch protection is configured).

Results are posted as a comment on the PR with the full quality report.

### Claude CLI authentication

The CI uses Claude CLI (`claude -p`) in non-interactive print mode. Authentication is handled via the `CLAUDE_CREDENTIALS` repository secret, which contains the JSON credentials from your local Claude CLI installation.

The model is configurable via the `KG_TEST_MODEL` repository variable (default: `claude-opus-4-6`).

### Refreshing the credentials

The Claude CLI token expires periodically. When CI starts failing with `Claude CLI exited with code 1`, refresh the secret:

1. Copy your local credentials to clipboard:

   ```bash
   security find-generic-password -s "Claude Code-credentials" -a "$(whoami)" -w | pbcopy
   echo "Token copied to clipboard"
   ```

2. Go to **GitHub > Repository > Settings > Secrets and variables > Actions**

3. Update the `CLAUDE_CREDENTIALS` secret with the clipboard contents

4. Re-run the failed workflow
