## Analisi Critica del Piano Attuale

### Cosa funziona bene nel piano esistente

1. **GCP + Cloud Run** — scelta corretta per scale-to-zero, semplicita, costi
2. **Firebase Hosting** per SPA — SPA fallback gratuito, nessun load balancer
3. **Neo4j su GCE VM** con backup su GCS — pragmatico per fase iniziale
4. **SQLite → Cloud SQL PostgreSQL** — necessario, ben pianificato
5. **CV storage → GCS** — rimuovere Fernet per i file, tenere SSE

### Gap critici identificati

#### GAP 1: Claude CLI non funziona in produzione (BLOCCANTE)

**Problema:** Il CV processing chiama `claude -p` via subprocess (`claude_classifier.py:41`). Il CLI richiede un login interattivo e credenziali locali — non funziona in un container Docker.

**Soluzione:** Migrare a **Vertex AI** (Claude su GCP) tramite l'SDK Anthropic Python con backend Vertex. Dettagli nella sezione dedicata sotto.

#### GAP 2: Note enhancement usa lo stesso CLI

`notes/router.py` chiama lo stesso `call_claude()`. Anche questo deve migrare a Vertex AI.

#### GAP 3: Rate limiting per-instance

SlowAPI usa counter in-memory. Con N istanze Cloud Run, i limiti diventano `3 × N` per utente. Il piano lo menziona come "accettabile" ma non da un trigger chiaro per il fix.

**Trigger consigliato:** quando Cloud Run scala stabilmente a 2+ istanze (monitorabile via Cloud Monitoring metric `run.googleapis.com/container/instance_count`), aggiungere Memorystore Redis (~$7/mese).

#### GAP 4: Nessuna strategia per cold start

La pipeline LLM (Vertex AI) aggiunge latenza. Il primo request dopo scale-to-zero colpisce:
1. Container startup (~2-5s)
2. Neo4j driver initialization + probe query (~1-2s)
3. Se e un CV upload: Vertex AI call (~30-120s)

**Mitigazione:** `min-instances: 1` in staging/produzione (costo: ~$5-10/mese extra). L'endpoint LLM ha gia timeout di 300s, quindi e compatibile.

#### GAP 5: Embedding placeholder

I vector index (1536 dim, cosine) sono in place ma gli embedding sono placeholder (SHA-512 hash). Il piano non menziona come attivarli.

**Nota:** Non bloccante per il deploy. Vertex AI supporta anche `text-embedding` models — puo essere aggiunto in fase successiva senza cambi architetturali.

## Vertex AI — Claude per CV Processing

### Perche Vertex AI

| Aspetto | Claude CLI (attuale) | Anthropic API diretta | Vertex AI |
|---------|---------------------|----------------------|-----------|
| Funziona in container | No | Si | Si |
| Autenticazione | Login interattivo | API key statica | Workload Identity (no chiavi) |
| Billing | Subscription personale | Conto Anthropic separato | Fatturato su GCP (unico billing) |
| Modelli disponibili | Tutti | Tutti | Claude Opus, Gemini Pro (nativo GCP) |
| Regione dati | US (default) | US/EU | Selezionabile (europe-west1) |
| Integrazione GCP | Nessuna | Nessuna | Nativa (IAM, logging, quotas) |

**Decisione: Vertex AI** — billing unificato GCP, nessuna API key da gestire, residenza dati EU, integrazione nativa con Cloud Run via Workload Identity.

### Modelli da usare

| Uso | Modello | Provider | Model ID |
|-----|---------|----------|----------|
| CV extraction (primario) | Claude Opus | Anthropic via Vertex AI | `claude-opus-4@20250514` |
| CV extraction (fallback) | Gemini 2.5 Pro | Google nativo | `gemini-2.5-pro` |
| Note enhancement | Claude Opus | Anthropic via Vertex AI | `claude-opus-4@20250514` |

**Nessun fallback rule-based, nessun Ollama, nessun Sonnet.** La catena e:
1. Claude Opus (primario) — qualita massima, composite score 0.940
2. Gemini 2.5 Pro (fallback) — modello top Google, nativo su Vertex AI, nessun costo aggiuntivo di intermediazione

> I model ID Claude su Vertex AI seguono il formato `claude-{tier}-{version}@{date}`. Gemini usa `gemini-2.5-pro` (o `gemini-2.5-pro-preview-05-06` per la versione specifica). Verificare disponibilita con `gcloud ai models list --region=europe-west1`.

#### Nuova firma di `call_claude()` (primario)

```python
async def call_claude(
    system_prompt: str,
    user_message: str,
    model: str | None = None,
    timeout: int | None = None,
) -> dict:
    """Call Claude Opus via Vertex AI and return response with usage metadata."""
    from app.config import settings

    if timeout is None:
        timeout = settings.llm_timeout_seconds

    client = AnthropicVertex(
        region=settings.vertex_region,
        project_id=settings.gcp_project_id,
    )

    if model is None:
        model = settings.claude_model  # "claude-opus-4@20250514"

    message = await asyncio.to_thread(
        client.messages.create,
        model=model,
        max_tokens=8192,
        system=system_prompt,
        messages=[{"role": "user", "content": user_message}],
    )

    return {
        "content": message.content[0].text,
        "cost_usd": None,  # Vertex AI billing is via GCP, not per-call
        "duration_ms": None,
        "input_tokens": message.usage.input_tokens,
        "output_tokens": message.usage.output_tokens,
    }
```

#### Nuova `call_gemini()` (fallback)

```python
async def call_gemini(
    system_prompt: str,
    user_message: str,
    model: str | None = None,
    timeout: int | None = None,
) -> dict:
    """Call Gemini Pro via Vertex AI as fallback provider."""
    from app.config import settings
    import google.generativeai as genai

    if timeout is None:
        timeout = settings.llm_timeout_seconds

    if model is None:
        model = settings.gemini_model  # "gemini-2.5-pro"

    gemini_model = genai.GenerativeModel(model)

    response = await asyncio.to_thread(
        gemini_model.generate_content,
        contents=[user_message],
        generation_config={"max_output_tokens": 8192},
        system_instruction=system_prompt,
    )

    return {
        "content": response.text,
        "cost_usd": None,
        "duration_ms": None,
        "input_tokens": response.usage_metadata.prompt_token_count if response.usage_metadata else None,
        "output_tokens": response.usage_metadata.candidates_token_count if response.usage_metadata else None,
    }
```

> Nota: entrambe le funzioni usano `asyncio.to_thread` perche i SDK sync di Anthropic e Google non sono nativamente async. In alternativa: `AsyncAnthropicVertex` per Claude, `genai.GenerativeModel.generate_content_async()` per Gemini.

#### Fallback chain

```
# Produzione (unica configurazione)
LLM_FALLBACK_CHAIN=claude-opus,gemini-pro

# Development (locale — stesso comportamento, usa Vertex AI se configurato)
LLM_FALLBACK_CHAIN=claude-opus,gemini-pro
```

**Nessun rule-based, nessun Ollama, nessun Sonnet.** Se Opus fallisce (timeout, errore API, quota), si passa a Gemini Pro. Se anche Gemini fallisce, l'errore viene propagato all'utente — meglio un errore esplicito che un'estrazione di bassa qualita.

### IAM per Vertex AI

Il service account di Cloud Run necessita del ruolo:

```
roles/aiplatform.user
```

Questo permette di chiamare i modelli Vertex AI. Nessuna API key, nessun secret.

```bash
gcloud projects add-iam-policy-binding PROJECT_ID \
  --member="serviceAccount:orbis-api@PROJECT_ID.iam.gserviceaccount.com" \
  --role="roles/aiplatform.user"
```

### Abilitare i modelli su Vertex AI

#### Claude Opus (richiede console)

Claude non e abilitato di default. Richiede:

1. Andare su **Vertex AI > Model Garden** nella console GCP
2. Cercare "Claude" e abilitare **Claude Opus 4**
3. Accettare i Terms of Service di Anthropic
4. Attendere l'approvazione (puo richiedere qualche ora per Opus)

> L'abilitazione dei modelli Anthropic richiede la console — non e completamente automatizzabile via CLI.

#### Gemini Pro (nativo, nessuna abilitazione necessaria)

Gemini e il modello nativo di Google — e disponibile automaticamente una volta abilitata l'API `aiplatform.googleapis.com`. Nessun Terms of Service aggiuntivo.

## Networking e Sicurezza

### VPC

```bash
# Subnet dedicata (se non si usa default)
gcloud compute networks subnets create orbis-subnet \
  --network=default \
  --region=europe-west1 \
  --range=10.0.1.0/24

# VPC Access Connector (Cloud Run → VPC)
gcloud compute networks vpc-access connectors create orbis-connector \
  --region=europe-west1 \
  --subnet=orbis-subnet \
  --min-instances=2 \
  --max-instances=3 \
  --machine-type=e2-micro
```

### Firewall

```bash
# VPC connector → Neo4j Bolt
gcloud compute firewall-rules create allow-bolt-from-connector \
  --network=default \
  --allow=tcp:7687 \
  --source-ranges=10.0.1.0/24 \
  --target-tags=neo4j

# Bloccare accesso esterno a Neo4j (dovrebbe essere default senza IP pubblico)
# Cloud SQL non richiede firewall rule — usa Cloud SQL Auth Proxy
```

### Sicurezza

- **Nessuna API key statica** per LLM (Vertex AI via Workload Identity)
- **Nessuna API key statica** per GCS (Workload Identity)
- **Cloud SQL Auth Proxy** (built-in in Cloud Run, connessione Unix socket)
- **Neo4j** solo su IP privato, accessibile solo via VPC connector
- **Secret Manager** per tutti i secret, montati come env var in Cloud Run
- **Firebase Hosting** serve solo file statici — nessun secret lato frontend
- **MCP Server** non esposto pubblicamente (`--no-allow-unauthenticated`)

## Script di Deploy Automatizzato

### Struttura

```
infra/
  gcp/
    setup.sh            # Fase 0: crea tutte le risorse GCP
    deploy-backend.sh   # Deploy/aggiorna backend Cloud Run
    deploy-mcp.sh       # Deploy/aggiorna MCP Cloud Run
    deploy-frontend.sh  # Build + deploy Firebase Hosting
    teardown.sh         # Rimuovere tutto (con conferma)
    variables.env       # Variabili condivise (PROJECT_ID, REGION, etc.)
  neo4j/
    init.cypher         # (esistente)
    backup.sh           # Script di backup Neo4j → GCS
  postgres/
    init.sql            # Schema PostgreSQL
```

### `variables.env` (template)

```bash
# GCP Configuration
export PROJECT_ID="orbis-prod"
export REGION="europe-west1"
export ZONE="europe-west1-b"

# Naming
export BACKEND_SERVICE="orbis-api"
export MCP_SERVICE="orbis-mcp"
export SERVICE_ACCOUNT="orbis-api"
export VPC_CONNECTOR="orbis-connector"
export SUBNET="orbis-subnet"
export NEO4J_VM="orbis-neo4j"
export SQL_INSTANCE="orbis-db"
export GCS_BUCKET="orbis-cv-files"
export AR_REPO="orbis"
export FIREBASE_SITE="orbis-prod"

# Neo4j VM
export NEO4J_MACHINE_TYPE="e2-standard-4"
export NEO4J_DISK_SIZE="50GB"

# Cloud SQL
export SQL_TIER="db-f1-micro"
export SQL_STORAGE="10GB"
```

### `setup.sh` (schema)

```bash
#!/usr/bin/env bash
set -euo pipefail
source "$(dirname "$0")/variables.env"

echo "=== Orbis GCP Setup ==="

# 1. Abilita API
gcloud services enable run.googleapis.com compute.googleapis.com ...

# 2. Service Account + IAM
gcloud iam service-accounts create $SERVICE_ACCOUNT ...
gcloud projects add-iam-policy-binding $PROJECT_ID ...

# 3. VPC + Connector
gcloud compute networks subnets create $SUBNET ...
gcloud compute networks vpc-access connectors create $VPC_CONNECTOR ...

# 4. Neo4j VM
gcloud compute instances create $NEO4J_VM ...
# + startup script per installare Neo4j

# 5. Cloud SQL
gcloud sql instances create $SQL_INSTANCE ...
gcloud sql databases create orbis --instance=$SQL_INSTANCE

# 6. GCS
gcloud storage buckets create gs://$GCS_BUCKET ...

# 7. Secret Manager
# (interattivo — chiede i valori dei secrets)

# 8. Artifact Registry
gcloud artifacts repositories create $AR_REPO ...

# 9. Vertex AI — abilitazione modelli (richiede console per Terms of Service)
echo ">>> Abilitare Claude su Vertex AI Model Garden manualmente dalla console"

# 10. Firebase
firebase init hosting

# 11. Firewall
gcloud compute firewall-rules create allow-bolt-from-connector ...

echo "=== Setup completato ==="
```

## CI/CD Pipeline

### GitHub Actions — Deploy

```yaml
# .github/workflows/deploy.yml
name: Deploy to GCP

on:
  push:
    branches: [main]

jobs:
  deploy-backend:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      id-token: write
    steps:
      - uses: actions/checkout@v4
      - uses: google-github-actions/auth@v2
        with:
          workload_identity_provider: projects/PROJECT_NUM/locations/global/workloadIdentityPools/github/providers/github
          service_account: deployer@PROJECT_ID.iam.gserviceaccount.com
      - uses: google-github-actions/setup-gcloud@v2
      - name: Build and deploy backend
        run: |
          gcloud builds submit backend/ \
            --tag $REGION-docker.pkg.dev/$PROJECT_ID/$AR_REPO/api:$GITHUB_SHA
          gcloud run deploy orbis-api \
            --image $REGION-docker.pkg.dev/$PROJECT_ID/$AR_REPO/api:$GITHUB_SHA \
            --region $REGION \
            --service-account orbis-api@$PROJECT_ID.iam.gserviceaccount.com \
            --vpc-connector orbis-connector \
            --vpc-egress private-ranges-only \
            --add-cloudsql-instances $PROJECT_ID:$REGION:orbis-db \
            --set-secrets="JWT_SECRET=jwt-secret:latest,ENCRYPTION_KEY=encryption-key:latest,NEO4J_PASSWORD=neo4j-password:latest,DATABASE_URL=database-url:latest" \
            --set-env-vars="ENV=production,NEO4J_URI=bolt://10.0.1.X:7687,NEO4J_USER=neo4j,LLM_PROVIDER=vertex,LLM_FALLBACK_CHAIN=claude-opus:gemini-pro,VERTEX_REGION=$REGION,GCP_PROJECT_ID=$PROJECT_ID"

  deploy-frontend:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 20 }
      - run: cd frontend && npm ci && npm run build
      - uses: FirebaseExtended/action-hosting-deploy@v0
        with:
          repoToken: ${{ secrets.GITHUB_TOKEN }}
          firebaseServiceAccount: ${{ secrets.FIREBASE_SERVICE_ACCOUNT }}
          channelId: live

  smoke-test:
    needs: [deploy-backend, deploy-frontend]
    runs-on: ubuntu-latest
    steps:
      - run: |
          curl -sf https://orbis-api-xxxxx.run.app/health/ready
          curl -sf https://app.openorbis.com/
```

## Note Operative

### Primo deploy — ordine operazioni

1. `setup.sh` — crea infrastruttura (una volta sola)
2. Abilitare Claude Opus su Vertex AI Model Garden (manuale, Terms of Service Anthropic)
3. Merge del codice migrato (Fase 1-3) su `main`
4. `deploy-backend.sh` — prima build + deploy backend
5. `deploy-frontend.sh` — build + deploy frontend
6. Verificare health check, testare upload CV
7. Eseguire migration dati (se ci sono dati da migrare)

### Rollback

- **Cloud Run**: `gcloud run services update-traffic orbis-api --to-revisions=PREVIOUS_REVISION=100`
- **Firebase Hosting**: `firebase hosting:rollback`
- **Database**: Cloud SQL backup automatici + Neo4j dump su GCS

### Monitoring

- **Cloud Logging**: log JSON strutturati dal backend
- **Cloud Monitoring**: alerting su 5xx, latenza p99, instance count
- **Vertex AI dashboard**: token usage, costi LLM, latenza modelli
